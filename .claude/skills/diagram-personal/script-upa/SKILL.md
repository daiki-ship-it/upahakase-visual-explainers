# Script Upa

テーマから**パニっくん×ウパ博士の対話台本**（`dialogue.md`）を生成し、diagram-upa スキルに渡せる状態にする。

**完了条件**: `output/{プロジェクト名}/dialogue.md` が生成され、diagram-upa スキルで図解を作れる状態になること。

## 依存

| 内容 | パス（リポジトリルートから） |
|------|------------------------------|
| **台本フォーマット仕様（唯一の正本）** | `.claude/skills/diagram-personal/script-upa/references/dialogue-format.md` |
| **キャラ設定・トーン・セリフの書き方** | `.claude/skills/diagram-personal/diagram-upa/references/character-usage.md` |
| **用語のやさしい言い換え** | `.claude/skills/diagram-personal/diagram-upa/references/term-dictionary.md` |
| **完成見本台本（フォーマットの参照用）** | `output/ai-daily-report-slack-apr2026/dialogue.md` |

## ワークフロー

全体は **入力収集 → 第1稿生成 → 専門家レビュー → 台本修正① → 読者視点レビュー → 台本磨き上げ → 出力** である。

### 1. 入力収集

以下の5項目をユーザーから確認してから第1稿を生成する。未提供の項目は質問する。

1. **テーマ・題材** — 何について話すか
2. **主メッセージ**（1文） — 読後に読者が言えるようになること
3. **章数の目安** — 短め（3-4章）/ 標準（5-6章）/ 長め（7章以上）
4. **参考URL・資料** — 任意。あれば `WebFetch` 等で取得して補強に使う
5. **読者の前提知識** — デフォルト：パニっくんと同レベル（IT苦手な非エンジニア）

主メッセージの1文が定まらない場合は、テーマから一緒に言語化する。

### 2. 第1稿生成

- [references/dialogue-format.md](references/dialogue-format.md) の仕様に従って台本第1稿を生成する
- キャラ設定・トーン・セリフの書き方は `character-usage.md` に従う
- 用語のやさしい言い換えは `term-dictionary.md` を参照し、**ウパ博士のセリフの中**に取り込む（独立した用語ブロックは作らない）
- パニっくんの疑問は「読者が自然に抱く疑問の順序」に従い、わざとらしくならないようにする

### 3. 専門家レビュー（サブエージェント）

**generalPurpose** サブエージェントで、台本に現れる AI・技術・製品・数値・時期・バージョン・固有名詞などを最新情報に照らして検証する。WebSearch を積極的に使用する。

```
Task({
  subagent_type: "generalPurpose",
  description: "台本内AI・技術情報の専門家レビュー",
  prompt: `
    あなたはAI・テクノロジー分野の専門家です。以下の台本に登場する、
    AI・機械学習・クラウド・開発ツール・製品名・統計・日付・バージョンなど、
    事実確認が必要な記述を列挙してください。
    公的ドキュメント・一次情報・権威ある解説などを WebSearch で確認し、
    誤り・誇張・曖昧さ・時代錯誤があれば指摘し、修正案の方向性を短く示してください。
    台本に触れられていない論点を新たに「追加すべき」と提案する必要はありません（台本の忠実な検証が目的）。

    【台本】
    {台本全文またはファイルパス}

    出力は箇条書きで「確認した記述 / 結論 / 根拠の種類（公式・報道・解説など）」がわかるように整理してください。
  `
})
```

### 4. 台本修正①

- Phase 3 の結果を反映し、**誤り・リスクのある表現を修正**する
- 断定表現はリスクがあれば「条件付き」などトーンも調整する
- この時点の台本を、以降の読者視点レビューの**唯一のストーリー正**とする

### 5. 読者視点レビュー（サブエージェント）

**generalPurpose** サブエージェントで、パニっくんに近い読者視点で台本のわかりやすさ・違和感をレビューさせる。**セリフの順序・難しさ・たとえの噛み合い・情報量**を優先する。

```
Task({
  subagent_type: "generalPurpose",
  description: "読者視点の台本レビュー",
  readonly: true,
  prompt: `
    あなたは「パニっくん」に近い読者です。ITは苦手だが、AIには興味がある。
    専門用語の羅列や説明不足には敏感。
    レビューで「用語解説ボックスを足せ」「まず覚える3つを入れろ」など、
    対話以外の教材パーツ追加を提案しないでください。
    改善はセリフの中身・順序・前提の補足に限定する。

    【台本】
    {台本全文またはファイルパス}

    ## 観点
    1. どこで意味がつながらなくなるか（前提が抜けている箇所）
    2. 難しい言葉・一度に来る概念の多さ
    3. たとえ話の違和感・順序の悪さ
    4. パニっくんのセリフが読者らしくない箇所
    5. 読後に説明できるか（ゴールの明確さ）

    ## 出力形式
    ### 致命的（このままでは読者が置いていかれる）
    - ...
    ### 改善したい（直すと一気に良くなる）
    - ...
    ### 良い（残すべき）
    - ...
    ### 総評（短く）
  `
})
```

### 6. 台本磨き上げ

- Phase 5 のうち**致命的・改善したい**に対し、台本を改稿する。**良い**は維持する
- 改稿が不要と判断できる場合は Phase 4 の台本をそのまま最終版とする

### 7. 出力・引き渡し案内

- 最終台本を `output/{プロジェクト名}/dialogue.md` に保存する（プロジェクト名はテーマを元にした kebab-case、例: `ai-daily-report-slack-apr2026`）
- ユーザーへの報告は **ファイルパスと台本の概要（章立て・主メッセージ1文）** のみ
- 「**次に diagram-upa スキルを呼び出し、`{ファイルパス}` を台本として渡してください。**」と案内する
